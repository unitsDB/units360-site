Platform Overview
🚀 What is Units360?

Units360.ai is a low-code usage metering platform — the operating system for any business that sells by the unit. Instead of building custom billing infrastructure, you define your unit model once, and Units360 auto-provisions everything else.

In this demo, the unit is kWh (kilowatt-hour). But it works equally for tokens, API calls, ride credits, consultation hours — any measurable service.

The power provider logs into Units360, creates a Metadata Template (MDT) for kWh, and configures all their rate rules. Units360 then handles all the real-time metering, net balance tracking, and invoice generation automatically.

Net Metering: When the home's solar panels produce more than it consumes, the excess goes back to the grid — and the utility credits the customer. Units360 tracks both directions in real time using a single bidirectional unit.
Core Concept · 1 of 3
⚡ What is a Unit?

A Unit is the fundamental thing being measured and billed. It defines what you are counting, how it accumulates, and when it resets.

unit_idkWh — the unique identifier
display_nameKilowatt Hour
directionBIDIRECTIONAL — can go up (export) or down (import)
reset_periodMonthly — balance resets on the 1st of each month
precision3 decimal places (e.g., 0.045 kWh per reading)

Units can be unidirectional (tokens only consumed) or bidirectional (energy exported / imported). The same unit definition handles both directions — the sign of the transaction determines direction.

Core Concept · 2 of 3
🏠 What is an Account?

An Account is the entity whose usage is being tracked — a customer, a device, a site, or a subscription. Each account gets its own running balance of the unit, governed by its rate plan.

account_idRES-CA-94102 — unique account identifier
typeResidential Solar Customer
rate_planE-TOU-C — Time-of-Use, with peak pricing 8–10 PM
solar_exportENABLED — Net Metering program active
peak_hours8:00 PM – 10:00 PM daily
current_balanceLive kWh net balance — updated per event

When a CloudEvent arrives, Units360 looks up the account, applies its rate plan rules, and updates its balance atomically. Multiple accounts can share the same unit definition but have different rate plans.

Core Concept · 3 of 3
📋 What is a Transaction?

A Transaction is a single metering event — one reading from the smart meter that says "this account consumed or exported X kWh right now." Transactions are the atomic records that build up the running balance.

quantity0.0105 kWh — the amount in this reading
directionEXPORT (positive) or IMPORT (negative)
price_applied$0.0842/kWh export · $0.1240/kWh import · +$0.18 peak
balance_afterRunning net total updated after each event
timestampISO 8601 UTC — every event is time-stamped precisely

Transaction rules — set up in the MDT — define how to price each event based on direction, time of day, and account plan. Units360 applies these rules automatically at ingestion time, with no custom code required.

At month end, all transactions are summed: exported kWh earn credits, imported kWh incur charges. The net produces the invoice automatically.
Configuration Layer
📐 MDT — Metadata Template

The MDT (Metadata Template) is Units360's low-code configuration builder. It's where the power provider defines everything about their metering model — with no code, no database schemas, no infrastructure setup.

Think of the MDT as a blueprint: define it once, and it governs every meter event, every balance update, and every invoice for that unit type.

Unit tabDefine the unit (kWh), direction, precision, and reset period
Account tabSet account types, rate plans, solar export eligibility, peak windows
Transaction tabConfigure pricing rules: export rate, import rate, peak surcharges

In the demo, you see the MDT form fill in during Act 1 (6 AM). That represents a power provider logging into Units360 and configuring their metering system in minutes — not months of custom development.

MDT outputs: Once saved, Units360 auto-generates a REST API endpoint, a CloudEvent schema for the smart meter, and a live analytics dashboard. No DevOps team required.
Runtime Engine
⚙️ RTE — Run Time Engine

The RTE (Run Time Engine) is the live processing layer that runs once an MDT is deployed. It continuously ingests meter events, applies rate rules, updates balances, and maintains the audit ledger — all in real time.

Event ingestionAccepts CloudEvents from smart meters via REST POST
Balance engineAtomic ledger update per event — bidirectional, millisecond-accurate
Rate applicationReads MDT rules; applies export, import, or peak pricing per event time
Threshold alertsFires webhooks when balance crosses configured thresholds
Invoice triggerAt period end, aggregates all transactions → auto-generates invoice JSON
Analytics feedLive data streamed to the auto-provisioned Superset dashboard

Event Protocol
📡 CloudEvents — The JSON Payload

Units360 uses the CloudEvents 1.0 spec — an open industry standard for describing event data. Every meter reading the smart meter sends is formatted as a CloudEvent JSON payload and POSTed to the Units360 RTE endpoint.

// Smart meter → Units360 RTE (one reading) POST /api/v1/events/RES-CA-94102 Content-Type: application/cloudevents+json { "specversion": "1.0", "type": "com.units360.energy.meter", "source": "solar/RES-CA-94102", "time": "2024-06-21T12:03:45Z", "data": { "unit": "kWh", "qty": 0.0105, "direction": "EXPORT", "balance": 42.7 } }
specversionAlways "1.0" — CloudEvents standard version
typeNamespaced event type — identifies the meter category
sourceDevice + account identifier — where the reading came from
timeExact UTC timestamp — used for TOU (time-of-use) pricing lookup
data.unitMust match the MDT unit_id — "kWh" here
data.qtyThe reading amount — RTE adds this to the account balance
data.directionEXPORT = solar generating; IMPORT = consuming from grid

The RTE validates the payload against the MDT schema, applies the correct rate, updates the balance atomically, and returns a 202 Accepted response in under 10ms — at scale, across millions of meters.
