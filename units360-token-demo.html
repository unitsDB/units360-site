Platform Overview
🚀 What is Units360?

Units360.ai is a low-code usage metering platform — the operating system for any business that sells by the unit. Instead of building custom billing infrastructure, you define your unit model once, and Units360 auto-provisions everything else.

In this demo, the unit is AI Token. But it works equally for kWh, API calls, ride credits, consultation hours — any measurable service.

Core Concept · 1 of 3
⚡ What is a Unit?

A Unit is the fundamental thing being measured and billed. It defines what you are counting, how it accumulates, and when it resets.

unit_idai-token — the unique identifier
display_nameAI Token
directionUNIDIRECTIONAL — tokens only consumed, never returned
reset_periodMonthly — balance resets on the 1st of each month
precision0 decimal places — tokens are whole numbers

Units can be unidirectional (tokens only consumed) or bidirectional (energy exported / imported). For AI tokens, direction is always inbound — each API call consumes tokens from the account's monthly allowance.

Core Concept · 2 of 3
👤 What is an Account?

An Account is the entity whose usage is being tracked — a customer, a tenant, a device, or a subscription. Each account gets its own running balance governed by its rate plan.

account_iddev-42 — unique developer tenant ID
planPro — 100,000 tokens/month included
overage_rate$0.002 per 1,000 tokens above the 100K limit
alert_threshold80% usage — webhook fired to notify the developer
hard_limit100,000 tokens — API blocked until next billing cycle
current_balanceLive token count — updated per API call

When an API call arrives, Units360 looks up the account, checks its balance against the rate plan, updates the running total atomically, and fires threshold alerts when configured limits are crossed.

Core Concept · 3 of 3
📋 What is a Transaction?

A Transaction is a single metering event — one API call that reports "this account consumed X tokens right now." Transactions are the atomic records that build the running balance.

quantity847 tokens — input + output tokens for this call
modelclaude-3-opus · gpt-4o · gemini-pro (any LLM)
featureautocomplete · code-review · debug — tagged per call type
balance_afterRunning total updated after each event
timestampISO 8601 UTC — every event is time-stamped precisely

Transaction rules — set up in your usage configuration — define how to price each event based on model tier, feature type, and account plan. Units360 applies these rules automatically at ingestion time, with no custom code required.

At month end, all transactions are summed. If the tenant exceeded 100K tokens, the overage is calculated and the invoice is generated automatically.
Configuration Layer
📐 Usage Configuration

The Usage Configuration is Units360's low-code setup builder. It's where the AI company defines everything about their metering model — with no code, no database schemas, no infrastructure setup.

Define it once, and it governs every API call, every balance update, and every invoice for that unit type.

Unit tabDefine the unit (ai-token), direction (inbound), reset period (monthly)
Account tabSet plan tiers, included quotas, hard limits, and overage rates
Transaction tabConfigure feature tagging, model-tier pricing, threshold webhooks
Once saved, Units360 auto-generates a REST API endpoint, a CloudEvent schema for your AI gateway, and a live analytics dashboard. No DevOps team required.

Event Protocol
📡 CloudEvents — The JSON Payload

Units360 uses the CloudEvents 1.0 spec — an open industry standard for describing event data. Every API call your AI gateway makes is formatted as a CloudEvent JSON payload and POSTed to the Units360 live endpoint.

// AI gateway → Units360 (one API call) POST /api/v1/events/dev-42 Content-Type: application/cloudevents+json { "specversion": "1.0", "type": "com.units360.ai.token", "source": "gateway/dev-42", "time": "2024-06-21T14:03:45Z", "data": { "unit": "ai-token", "qty": 847, "model": "claude-3-opus", "feature": "autocomplete" } }
specversionAlways "1.0" — CloudEvents standard version
typeNamespaced event type — identifies the metering category
sourceGateway + account ID — where the call originated
data.unitMust match the configured unit_id — "ai-token" here
data.qtyTotal tokens for this call — RTE adds this to the balance
data.modelLLM used — enables per-model cost breakdown in analytics
data.featureFeature tag — enables per-feature usage reporting
